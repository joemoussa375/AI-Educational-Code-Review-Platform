Abstract—The rapid expansion of computer engineering curricula has exacerbated the student-to-instructor ratio, creating a critical bottleneck in delivering timely, personalized programming feedback. While students increasingly rely on Large Language Models (LLMs), these generalized tools are optimized for industrial code generation rather than pedagogical instruction, often resulting in hallucinated syntax and the circumvention of academic standards. This paper presents an AI-Powered Educational Feedback Loop Platform designed to transition automated review from a "code solver" to a rigorous "code tutor." We propose a hybrid architecture that integrates the heuristic reasoning of Qwen2.5-Coder-7B with deterministic static analysis via Pylint, ensuring the identification of complex logical vulnerabilities without syntax hallucination. To enforce strict pedagogical compliance and prevent prompt injection, the system utilizes a secure, offline Retrieval-Augmented Generation (RAG) engine grounded in PEP 8 and OWASP 2025 guidelines. By leveraging 4-bit quantization, the platform is optimized to operate entirely on edge-deployable, consumer-tier university hardware, guaranteeing data privacy and eliminating external API dependencies. The architecture is evaluated against a novel, three-tier 100-sample dataset designed to stress-test functional error recall, Abstract Syntax Tree (AST) validity, and operational context window limits. The results validate the platform as a scalable, secure, and highly accurate automated tutoring solution for modern engineering education.
Keywords—Automated Code Review, Large Language Models (LLMs), Retrieval-Augmented Generation (RAG), Static Code Analysis, Engineering Education, Model Quantization.
I.	INTRODUCTION
The rapid expansion of computer science and engineering curricula has created a significant pedagogical bottleneck: the demand for immediate, personalized programming feedback far outpaces the available academic resources. Programming mastery requires iterative practice and continuous code review, a task traditionally reliant on teaching assistants and professors. However, growing student-to-instructor ratios often result in delayed feedback loops, missing the critical window where learning is most effective. As noted by Mendonça et al. [1], automating this evaluation process is no longer just an experimental luxury but an educational necessity to maintain academic rigor at scale.
To bridge this feedback gap, students increasingly turn to out-of-the-box Large Language Models (LLMs). While highly capable, these generalized AI tools are structurally optimized for industrial productivity by generating working code as quickly as possible, rather than pedagogical support, which focuses on guiding students through core concepts and best practices [2]. This misalignment creates a phenomenon of "blind trust" among novice developers. Furthermore, pure LLMs inherently rely on probabilistic text generation, making them prone to hallucinations, such as inventing non-existent syntax errors or silently ignoring critical style guidelines [3]. Consequently, deploying unconstrained LLMs in an educational setting risks reinforcing poor programming habits rather than correcting them.
This paper proposes a paradigm shift from basic AI "code solvers" to an integrated Educational Feedback Loop Platform. We introduce a hybrid code review architecture that synergizes the probabilistic reasoning of a state-of-the-art open-weights model (Qwen2.5-Coder-7B) with the deterministic accuracy of static code analysis (Pylint). To eliminate hallucinations and enforce strict academic standards, the system integrates an offline Retrieval-Augmented Generation (RAG) knowledge base, securely grounding the AI's feedback in official documentation, including PEP 8 for style, SOLID for architecture, and OWASP for security guidelines. Optimized via 4-bit quantization, this architecture is specifically designed for secure, local deployment on consumer-tier university hardware, ensuring student data privacy while democratizing access to high-quality, continuous code review.
II.	BACKGROUND AND RELATED WORK
A.	The Shift to Hybrid Code Analysis
The limitations of relying solely on Large Language Models for automated code review have become increasingly evident in recent software engineering literature. While LLMs excel at semantic reasoning and natural language explanation, they lack the strict deterministic rule enforcement required for precise syntax validation. To counteract this, current research advocates for Hybrid Architectures—systems that fuse the heuristic strengths of generative AI with the factual precision of Static Analysis Tools (SATs). As demonstrated by the LegacyGuard framework [4], feeding the explicit error logs of a static analyzer directly into an LLM's prompt context transforms the AI from a flawed "bug detector" into a highly accurate "bug interpreter." This dual-engine strategy ensures that syntax gating remains strictly deterministic, while the LLM is reserved for explaining the logical intent behind the errors.
B.	Retrieval-Augmented Generation for Compliance
Beyond functional correctness, educational tools must rigorously enforce industry coding standards to prevent the perpetuation of poor stylistic habits. Generic, out-of-the-box LLMs tend to treat styling frameworks (such as Python's PEP 8) as flexible guidelines rather than strict operational constraints. To address this compliance gap, researchers have successfully implemented Retrieval-Augmented Generation (RAG) within automated review pipelines [5]. By forcing the model to access a localized vector database of specific documentation prior to generating a review, a RAG-enhanced system actively overrides the model's parametric memory. This ensures that the generated feedback is not an amalgamation of generic internet training data, but a compliant reflection of explicitly defined domain rules, a critical necessity for maintaining pedagogical trust.
III.	SYSTEM ARCHITECTURE AND METHODOLOGY
The proposed Educational Feedback Loop Platform transitions away from traditional end-to-end LLM processing in favor of a multi-stage, hybrid pipeline as seen in Fig. 1. The architecture processes inputs sequentially through a deterministic gatekeeper and a semantic retrieval engine before passing the finalized context to the generative inference model.

A.	Generative Inference Engine and Hardware Optimization
To achieve state-of-the-art coding and reasoning capabilities, the system utilizes the Qwen2.5-Coder-7B-Instruct model [8]. While enterprise-grade models typically require massive computational overhead and reliance on third-party cloud APIs, querying external APIs presents significant data sovereignty and privacy risks for educational institutions. To enable secure, local deployment on consumer-tier university hardware, the model is loaded utilizing a 4-bit Normal Float (NF4) quantization strategy via the QLoRA framework [9].
This quantization mathematically compresses the model's memory footprint from approximately 16 GB of VRAM down to 5 GB while retaining nearly full heuristic capability. Consequently, the entire architecture can be hosted locally on standard engineering lab environments or free-tier cloud infrastructure (such as Kaggle's dual-T4 GPUs) without the unpredictable costs of API-based token taxation. To further optimize the user experience, the heavy 4.5 GB quantized model weights are eager-loaded during the server startup phase, entirely removing load times from the student's web request cycle to prevent browser timeouts.
B.	Deterministic Syntax Gatekeeper
Because probabilistic LLMs struggle with exact syntax validation, a dedicated static analysis module acts as a factual gatekeeper before generative inference begins. User-submitted Python code is routed through a localized Pylint subprocess. This module performs critical pre-filtering: if the code contains an overwhelming number of syntax violations, the engine aggressively truncates the noise to prevent overflowing the LLM’s context window. The raw JSON output is then parsed into strict, human-readable facts (e.g., "Line 4: Missing module docstring"), which are injected into the final prompt as absolute realities rather than suggestions.
C.	Secure Offline Knowledge Vault (RAG Engine)
Live web-scraping for context retrieval introduces severe operational vulnerabilities, including extensive boot latencies, network timeouts, and exposure to web-based prompt injection attacks [6]. To secure the pipeline and ensure architectural compliance, the system utilizes an entirely offline Retrieval-Augmented Generation (RAG) knowledge vault.
External domain intelligence is anchored in a local Markdown database comprising strict pedagogical rulesets: the official PEP 8 Python Style Guide, the OWASP 2025 Security Guidelines, and core SOLID Design Principles. The retrieval engine utilizes a hybrid approach to extract relevant rules. A semantic search utilizing the FAISS library and the all-MiniLM-L6-v2 embedding model retrieves conceptually similar documentation, while a BM25 algorithm conducts exact-keyword matching. This dual-search methodology ensures that niche variables and specific function conventions are never overlooked by the vector similarity search.
D.	Pedagogical Prompt Engineering
To eliminate the unstructured hallucinations common in generic LLM interactions, the system employs a two-stage Structured Chain-of-Thought (SCoT) prompting methodology [7]. The backend discards legacy instruction formatting in favor of strict {"role": "system"} chat templates.
In the first stage, the model acts as a "Technical Analyst," rapidly parsing the student's code to generate distinct search queries for the offline RAG engine. In the second stage, the model acts as a "Senior Architect." The system aggregates the deterministic Pylint facts and the retrieved RAG domain rules into a single context prompt. The model is forced via penalty constraints to bypass repetitive degenerate loops and output feedback in exactly three sections:
1)	Critical Issues: Identification of logical bugs, algorithmic inefficiencies, and active safety hazards.
2)	Style Analysis: Explicit verification against the retrieved PEP 8 rules.
3)	Refactored Solution: A robust, pythonic, and safe correction of the original code, provided not just as an answer, but as an applied pedagogical example.
IV.	EXPERIMENTAL SETUP
To validate the efficacy, accuracy, and computational viability of hybrid architecture, a comprehensive evaluation framework was designed. Due to the lack of standardized open-source benchmarks specifically targeting pedagogical feedback and student-level programming errors, a custom Curated Tiered Dataset was engineered for this study.
A.	Computational Environment
All experimental evaluations were conducted within a cloud-simulated environment designed to mirror the resource constraints of consumer-tier, edge-deployable university hardware. The testbed utilized a Kaggle/Colab notebook equipped with dual NVIDIA Tesla T4 GPUs (16 GB VRAM capacity) and 31.4 GB of system RAM. The Qwen2.5-Coder-7B-Instruct engine was executed using 4-bit Normal Float (NF4) quantization via PyTorch, effectively reducing the model's memory footprint to approximately 5 GB and reserving the remaining VRAM for the Key-Value (KV) Cache required during extensive context window operations.
B.	Dataset Construction
The evaluation dataset comprised 100 synthetically generated Python scripts containing deterministic flaws across varying levels of complexity. The dataset was stratified into three distinct tiers to stress-test specific architectural components:
•	Tier 1: Foundational Snippets (50 Samples): Short scripts (10–50 lines) focusing on common novice mistakes, including PEP 8 style violations, off-by-one logical errors, and mutable default arguments. Designed to test the baseline retrieval accuracy of the RAG engine and Pylint syntax detection.
•	Tier 2: Intermediate Modules (30 Samples): Mid-sized files (100–300 lines) containing interconnected classes, inefficient algorithms (e.g., nested O(n²) loops), and security hazards like unclosed file streams and eval() vulnerabilities.
•	Tier 3: Monolithic Structures (20 Samples): Massive scripts (500–1,000+ lines) designed specifically to stress-test the operational context window limits of the 16 GB hardware, monitoring token degradation, latency spikes, and KV cache exhaustion.
C.	Evaluation Metrics
To eliminate human bias during the evaluation phase, the system’s output was graded using an automated Python evaluation. Each of the 100 samples in the tiered dataset was paired with a Ground Truth JSON object containing an expected field. This field explicitly defined the primary flaw the script was designed to test (e.g., "Critical: Mutable default argument").
The system’s output was graded using three automated academic metrics:
a)	Functional Error Recall (F1-Score): The precision and recall of the model in identifying the specific bug defined in the Ground Truth JSON.
b)	Code Resolution Rate (AST Validity): The percentage of AI-refactored code blocks that parse into a valid Abstract Syntax Tree (AST), proving the model generates compilable, non-hallucinated syntax.
c)	Specification Compliance: The rate at which the generated feedback explicitly references and enforces the injected PEP 8 and OWASP RAG guidelines.
It is critical to note the strict string-matching logic of the evaluation script for calculating the F1-Score. The script operates as a rigid evaluator; if an AI model flagged a legitimate code anomaly (such as a missing docstring) that was not explicitly listed in that specific test case's expected ground truth, the script automatically penalized the model by recording a False Positive. This strict, isolated grading mechanism was intentionally designed to expose the behavioral differences between heuristic baseline models and hyper-compliant pedagogical architectures.

VI.	CONCLUSION AND FUTURE WORK
A.	Conclusion
This paper presented a novel Educational Feedback Loop Platform that successfully bridges the scalability gap in computer engineering education. By transitioning away from generic, black-box LLM interactions, the proposed hybrid architecture ensures that automated code review acts as a rigorous pedagogical tutor rather than a simple code solver. The integration of deterministic static analysis via Pylint with the advanced heuristic reasoning of the Qwen2.5-Coder-7B model creates a system capable of identifying complex logical vulnerabilities without hallucinating syntax errors. Furthermore, the implementation of a secure, offline Retrieval-Augmented Generation (RAG) engine successfully anchors the model's feedback in established industry standards, enforcing strict adherence to PEP 8 and OWASP guidelines. Crucially, by leveraging 4-bit quantization, the platform demonstrates that high-fidelity AI evaluation can be securely and locally deployed on consumer-tier university hardware, mitigating the privacy risks and unpredictable costs associated with external enterprise APIs. Ultimately, this architecture establishes a vital educational synergy: students receive instant, actionable feedback rather than facing the delays inherent to high student-to-tutor ratios, while teaching staff are freed from repetitive syntax debugging to focus their valuable time on higher-level instruction and mentorship.
B.	Future Work
While the current architecture successfully establishes the Student Interface for immediate submission feedback, the current vision encompasses a dual-platform approach. Future development will focus on the construction of a comprehensive Teaching Staff Dashboard. This interface will aggregate the data generated by the hybrid AI pipeline, allowing teaching assistants and professors to monitor class-wide performance trends, identify frequently violated coding principles, and flag struggling students early in the semester. Additionally, to address the hardware-constrained context window limitations observed during the monolithic script stress tests, future iterations will explore dynamic code-chunking algorithms to safely process exceptionally large student repositories without triggering memory exhaustion. Finally, to maximize the platform's utility across the broader engineering curriculum, future work will expand the deterministic analysis engine and RAG knowledge base to support additional programming languages widely utilized by students, specifically MATLAB and Java.
